# Turso And R2 Migration Design

## Goal

Reduce recurring Neon/Vercel Postgres cost risk by moving structured app data to Turso/libSQL first, then moving file storage from Vercel Blob to Cloudflare R2 in a later phase.

The final steady state is:

- Turso/libSQL is the source of truth for songs, contis, presets, sheet music metadata, export metadata, and generated image metadata.
- Cloudflare R2 is the source of truth for uploaded sheet music files, generated page images, exported PDFs, and longer-term file archives.
- Neon is removed from the production app after migration verification.
- Vercel Blob is faded out after existing files are copied to R2 and references are verified.

## Current Context

The app currently stores structured data in Neon/Vercel Postgres through Drizzle ORM:

- `lib/db/index.ts` uses `drizzle-orm/neon-http` and `POSTGRES_URL`.
- `lib/db/schema.ts` defines relational tables for songs, sheet music metadata, presets, contis, conti songs, PDF exports, generated song page images, and older Discord state.
- Server actions and queries import `db` directly from `@/lib/db`.
- Vercel Blob stores actual sheet music files, generated page images, and exported PDFs.
- Current measured file storage is small: 135 listed Blob objects and about 61.15 MB total. Original sheet music is 30 files and about 12.95 MB.

The cost issue came from Neon compute being kept active by frequent work, not from large storage volume. The highest-value first move is therefore replacing Neon with a storage model that does not bill by active compute time.

## Chosen Approach

Use a phased migration:

1. **Phase 1: Turso first**
   - Add a repository boundary around app data access.
   - Implement a Turso/libSQL-backed repository.
   - Keep a Neon-backed repository only for migration verification.
   - Compare Neon and Turso data during development and pre-cutover checks.
   - Switch production reads/writes to Turso only after verification passes.
   - Remove Neon env vars, imports, and dependencies after a short observation window.

2. **Phase 2: R2 next**
   - Add a `FileStorage` boundary around file upload/delete/list operations.
   - Keep the Vercel Blob implementation for existing file reads and fallback during migration.
   - Add a Cloudflare R2 implementation.
   - Send new uploads and generated files to R2 once the storage boundary is in place.
   - Copy existing Vercel Blob objects to R2 in batches.
   - Verify size/checksum/reference matches before updating metadata references.
   - Remove Vercel Blob usage after all references point to R2.

3. **Phase 3: Admin migration console**
   - Add an authenticated admin page for migration status, diffs, readiness checks, and cleanup actions.
   - Use this console for verification and operational visibility, not as a replacement for normal song/conti editing screens.

This avoids changing the database and file backend in one step. The first phase addresses the billing risk directly; the second phase improves long-term archive cost and portability.

## Non-Goals

- Do not build a Blob JSON database as the final source of truth.
- Do not leave dual-write behavior as a permanent production feature.
- Do not replace the existing song and conti editing UX with an admin JSON editor.
- Do not migrate Discord/Google Sheets automation behavior unless it touches repository boundaries needed for the storage migration.
- Do not copy binary files into Turso. Turso stores metadata and references only.

## Architecture

### Data Repository Boundary

Create a repository layer under `lib/repositories` or an equivalent local pattern:

```text
UI / Server Actions / Queries
        |
        v
StoryboardRepository
        |
        +-- TursoStoryboardRepository   final structured data backend
        +-- NeonStoryboardRepository    migration verification backend
        +-- MigrationVerifier           count/hash/sample diff checks
```

The initial repository surface should match existing app workflows rather than expose raw tables. It should include operations for:

- Listing and reading songs.
- Creating, updating, deleting, and searching songs.
- Listing, reading, creating, updating, deleting, and reordering sheet music metadata.
- Listing and reading contis with summaries.
- Reading a conti with songs and presets.
- Creating, updating, deleting, and reordering conti songs.
- Managing song presets and preset sheet music selections.
- Managing conti PDF export metadata.
- Managing generated song page image metadata.

Repository methods should preserve the existing server action return patterns where actions return `{ success, error?, data? }`.

### Turso Schema

Turso/libSQL should preserve the current relational model as closely as possible:

- Keep text primary keys generated by the app.
- Keep foreign key relationships where libSQL supports them.
- Keep JSON-like columns as text initially to reduce migration scope.
- Replace Postgres timestamp types with a Turso-friendly representation. ISO string timestamps are preferred because they are explicit, serializable, and easy to diff.
- Replace `ilike` behavior with normalized lowercase search fields or SQLite `lower(name) like lower(?)` style query helpers.

The Turso schema should intentionally mirror the current domain, not the old provider. This makes later cleanup possible without forcing a large UX rewrite.

### File Storage Boundary

Add a file storage layer:

```text
File APIs
        |
        v
FileStorage
        |
        +-- R2FileStorage          final file backend
        +-- VercelBlobFileStorage  existing file backend and migration fallback
```

The interface should cover:

- `putFile` for user uploads and generated exports.
- `deleteFile` for cleanup.
- `listFiles` for admin and migration checks.
- `copyFile` or equivalent migration helper.
- Metadata reporting: provider, key, URL, size, content type, checksum if available.

Structured metadata should store enough information to avoid being locked to public URLs only:

- `provider`: `vercel-blob` or `cloudflare-r2`
- `key`: storage object key
- `url`: current public or signed-access URL
- `fileName`
- `fileType`
- `size`
- `checksum` when available
- `createdAt`

Existing rows that only have `fileUrl` can be upgraded gradually by deriving provider/key during migration.

## Migration Verification

### Neon To Turso Verification

Verification should use several levels:

- Table/entity counts.
- Stable JSON serialization and hash per entity.
- Sample record comparisons for recent songs, contis, presets, sheet music, exports, and generated page images.
- Relationship checks:
  - Every conti song references an existing conti and song.
  - Every sheet music metadata record references an existing song.
  - Every preset references an existing song.
  - Every preset sheet music selection references existing preset and sheet music metadata.
  - Every PDF export references an existing conti.
  - Every generated song page image references existing song and conti.

During development, temporary dual-write may be used to compare behavior. This is a migration/testing mechanism only. The production target is a single Turso write path.

### Vercel Blob To R2 Verification

Verification should track:

- Vercel Blob object count and total bytes.
- R2 object count and total bytes.
- Per-object copied status.
- Size match.
- Checksum match when available.
- Metadata reference status: still Vercel Blob, copied to R2, switched to R2, or missing.

Because current measured storage is about 61.15 MB, the migration can be conservative and fully verified without meaningful storage cost pressure.

## Admin Migration Console

Add an authenticated admin route, proposed as `/admin/storage`.

The first version should include:

- **Overview**
  - Current data backend mode.
  - Current file backend mode.
  - Turso migration readiness.
  - R2 migration readiness.
  - Structured data counts.
  - File storage counts and total bytes.
  - Estimated time to hit relevant free-tier limits based on current growth.

- **Data Migration**
  - Neon export status.
  - Turso import status.
  - Neon vs Turso diff results.
  - Entity counts and hash summaries.
  - Readiness checklist for Neon fadeout.

- **File Migration**
  - Vercel Blob vs R2 object status.
  - Missing/copy-failed/mismatched files.
  - Batch copy action.
  - Batch metadata switch action after verification.
  - Readiness checklist for Vercel Blob fadeout.

- **Integrity Checks**
  - Orphan references.
  - Missing file objects.
  - Invalid provider/key/url metadata.
  - Checksum mismatch warnings.

- **Snapshots And Backups**
  - Export structured data snapshot.
  - Download migration reports.
  - Record cutover checkpoints.

Dangerous actions must require an explicit confirmation step. Examples include switching production backend mode, rewriting file metadata references, deleting old provider references, or marking a fadeout complete.

## Data Flow

### Phase 1 Cutover

1. Existing Neon data is exported through repository-aware scripts or admin actions.
2. The data is normalized for the Turso schema.
3. Turso tables are created and populated.
4. Verification compares Neon and Turso counts, hashes, and relationship checks.
5. Development and staging run against Turso.
6. Production backend mode switches to Turso.
7. Neon remains available for a short read-only observation window.
8. Neon env vars and provider-specific code are removed after confidence.

### Phase 2 File Cutover

1. `FileStorage` boundary is introduced while existing Vercel Blob reads continue.
2. R2 credentials and bucket config are added to Vercel env.
3. New uploads and generated files start writing to R2.
4. Admin migration copies existing Vercel Blob objects to R2.
5. Verification checks object size and checksum where possible.
6. Metadata references switch from Vercel Blob URLs to provider/key-aware R2 metadata.
7. Vercel Blob remains readable for a short fallback window.
8. Vercel Blob references and dependency usage are removed after readiness checks pass.

## Error Handling

- Repository errors should surface the existing Korean action error messages where user-facing behavior already exists.
- Migration verifier errors should be explicit and actionable: missing row, hash mismatch, missing object, size mismatch, or unsupported legacy metadata.
- Turso cutover should fail closed. If verification fails, do not switch production mode.
- R2 metadata switching should fail closed. If any copied object fails verification, do not rewrite references for that object.
- Admin dangerous actions should write a report before mutating state so failures can be audited.
- File delete operations should remain best-effort only after metadata has been safely updated or a snapshot has been taken.

## Testing Strategy

- Unit tests for repository mappers:
  - Neon row to domain object.
  - Domain object to Turso row.
  - Timestamp serialization.
  - JSON text column parsing/stringifying.
  - Case-insensitive song search helper.

- Unit tests for migration diff:
  - Equal datasets pass.
  - Missing entity fails.
  - Modified nested override data fails.
  - Orphan references fail.

- Unit tests for file storage metadata:
  - Vercel Blob URL provider/key derivation.
  - R2 key generation.
  - Missing size/checksum handling.

- Integration-style dry runs:
  - Export Neon data without mutating it.
  - Import into a local or preview Turso database.
  - Run verifier and inspect report.
  - Copy a small sample of Blob objects to R2 and verify metadata.

- Final validation before cutover:
  - `pnpm lint`
  - `pnpm build`
  - Existing focused test command from `package.json`
  - Admin readiness checks all passing

## Open Operational Notes

- Turso credentials, database URL, and auth token should be configured through Vercel env vars.
- R2 credentials, bucket name, account ID, public base URL, and optional custom domain should be configured through Vercel env vars.
- The migration should avoid logging secrets or full signed URLs.
- Existing untracked local analysis files should not be staged with the migration spec or implementation commits.

## Acceptance Criteria

- A design-approved implementation plan can be written from this spec without changing the migration direction.
- Phase 1 can be implemented independently from Phase 2.
- The final production source of truth for structured data is Turso, not Neon.
- The final production source of truth for files is R2, not Vercel Blob.
- Admin migration status is read-only by default and only performs mutations through explicit migration actions.
- There is no permanent production dual-write path after migration completes.
